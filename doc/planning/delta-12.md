# Delta 12a: Version management

**Status: DONE — 2026-05-03**

Add a single source of truth for the Tilr version, exposed via `tilr --version` on the CLI and a Help menu / About overlay in the app. Local development builds get a `-local-{git-hash}-{build-id}` suffix so it's always obvious whether someone is running a release build or a specific dev snapshot.

## Approach: Info.plist (no Swift recompilation)

The version lives in **`Info.plist`** for both targets, not in a generated Swift file. This avoids touching any source files when the version changes — every build can rotate the build id without invalidating Swift's incremental compile cache.

- The macOS app already has an `Info.plist` with `CFBundleShortVersionString`. We just stamp the dynamic version into that field.
- A macOS CLI binary can also embed an `Info.plist` directly into its executable via the linker (`__TEXT,__info_plist` section). xcodegen exposes this through the `info:` block on a target. Same `Bundle.main.infoDictionary` API works for both.
- A build-phase script writes the dynamic suffix into a build-setting file (xcconfig), and Xcode interpolates it into both Info.plists.

Net result: when version changes, only the Info.plist files are rewritten. **Zero Swift files are recompiled.**

## Format

```
release:    0.0.1
local dev:  0.0.1-local-afe4-b7c2         (git hash + build id)
```

The local suffix has two parts:
- **Git hash** (`afe4`): short hash of HEAD — tracks which commit we're based on
- **Build id** (`b7c2`): per-build random hex — distinguishes builds with the same source state

Why both? The git hash tells you the commit, but during dev iteration you often build many times without committing. The build id makes every build distinguishable in logs and version popups, so when the user says "I tested with `0.0.1-local-afe4-b7c2`", you know exactly which binary they ran.

Build id rotates every dev build (regenerated by the build phase). Release builds (`TILR_RELEASE=1`) skip the suffix entirely and produce just `0.0.1` — deterministic, identical between consecutive release builds at the same source state.

Alternative strategies considered and rejected:
- **`-dirty` suffix only** — doesn't distinguish iterations, all dirty builds look identical
- **Source content hash** — deterministic but expensive to compute every build
- **Monotonic counter file** — works but introduces stateful file outside git
- **Build timestamp** — works but harder to read/say than 4 hex chars

## Build phase script

`scripts/stamp-version.sh` runs from Xcode's build phase environment, where `$SRCROOT`, `$CONFIGURATION`, and `$MARKETING_VERSION` are set. It writes an xcconfig file that Xcode picks up as build settings:

```bash
#!/usr/bin/env bash
set -euo pipefail

base="${MARKETING_VERSION:-0.0.1}"
out="$SRCROOT/build/Generated/Version.xcconfig"

is_release="${TILR_RELEASE:-0}"
if [ "$is_release" = "1" ]; then
    suffix=""
else
    git_hash="$(cd "$SRCROOT" && git rev-parse --short=4 HEAD 2>/dev/null || echo nogit)"
    build_id="$(openssl rand -hex 2)"
    suffix="-local-${git_hash}-${build_id}"
fi

build_date="$(date -u +%Y-%m-%dT%H:%M:%SZ)"

mkdir -p "$(dirname "$out")"
cat > "$out" <<EOF
// AUTO-GENERATED — do not edit. Stamped by scripts/stamp-version.sh.
TILR_VERSION_SUFFIX = $suffix
TILR_BUILD_DATE = $build_date
EOF
```

Notes:
- xcconfig files don't quote string values; everything after `=` until newline is the value
- Xcode evaluates `$(MARKETING_VERSION)$(TILR_VERSION_SUFFIX)` at Info.plist generation time, producing the final version string
- The script always rewrites the xcconfig file. Since no Swift sources change, only Info.plist generation re-runs — extremely fast (<100ms for both targets combined)
- `--print` mode (when invoked manually outside the Xcode build phase) just prints the suffix without writing the file

## `Version.swift` shape

```swift
import Foundation

public enum Version {
    /// Full version string, e.g. "0.0.1-local-afe4-b7c2" or "0.0.1".
    public static var full: String {
        Bundle.main.infoDictionary?["CFBundleShortVersionString"] as? String ?? "unknown"
    }

    /// Base version without suffix, e.g. "0.0.1".
    public static var base: String {
        full.split(separator: "-").first.map(String.init) ?? full
    }

    /// Build timestamp from Info.plist (set by the build phase via xcconfig).
    public static var buildDate: String {
        Bundle.main.infoDictionary?["TILRBuildDate"] as? String ?? "unknown"
    }

    /// True if this is a local dev build.
    public static var isLocal: Bool { full.contains("-local-") }
}
```

For the build date to flow through, add `TILRBuildDate = $(TILR_BUILD_DATE)` to both targets' Info.plist properties in `project.yml`.

4. **Add `preBuildScripts` entry** to both targets in `project.yml`. Regenerate with `xcodegen`. Build. Confirm `Info.plist` for the app contains the dynamic version.
5. **Verify CLI Info.plist embedding**: dump the binary's plist with `otool -P` or `launchctl print` to confirm `CFBundleShortVersionString` is set.
6. **Create `Sources/Shared/Version.swift`** reading from `Bundle.main.infoDictionary`. Both targets already include `Sources/Shared`.
7. **Wire CLI `--version`**: add `version: Version.full` to the root `Tilr` `CommandConfiguration`. Test: `tilr --version` prints `0.0.1-local-afe4-b7c2`.
8. **Log on app launch**: add `TilrLogger.shared.log("Tilr v\(Version.full) starting", category: "app")` to `applicationDidFinishLaunching`.
9. **Build the About overlay**: copy `StatusOverlayWindow` as a starting template, strip space-listing logic, render the version block.
10. **Wire menu item**: extend `MenuBarController` with an "About Tilr…" item that opens the overlay.
11. **`.gitignore`** the generated xcconfig. **Add `just version`** recipe.

## Acceptance criteria

- `tilr --version` prints `0.0.1-local-{git-hash}-{build-id}` on a dev checkout (e.g. `0.0.1-local-afe4-b7c2`).
- Two consecutive `just build` invocations produce **different** build ids — the build id rotates every build.
- A clean `TILR_RELEASE=1` build stamps just `0.0.1` (no suffix). Two such builds at the same source state produce **identical** stamps.
- App startup log line shows the same string the user sees in the About overlay.
- Menu bar → "About Tilr…" displays an overlay with version, build date, bundle ID; dismisses on Esc.
- **No Swift file is recompiled** when only the version changes between builds — verified by checking Xcode's build log shows zero compile actions for `*.swift` files on a no-source-change rebuild.

## Notes / non-goals

- Not adding `tilr version` as a subcommand — `--version` (ArgumentParser convention) is sufficient and matches `git`, `cargo`, `gh` etc.
- Not signing/notarising release builds in this delta — that's still part of Delta 14.
- The CLI gets an embedded Info.plist via `__TEXT,__info_plist` (xcodegen handles the linker flag). Confirm with `otool -P /path/to/tilr` if needed.
- If the script can't find git (e.g. someone built from a tarball), the git hash component falls back to `nogit` (full stamp `0.0.1-local-nogit-{build-id}`).
